## 4. ISSUES FOUND

### 4.1 CRITICAL

#### C-1 — "arms dealer" missing from adverse-keywords → false-negative on OFAC-SDN subjects
**File:** `web/lib/data/adverse-keywords.ts:78-88`  
**Finding:** The `proliferation-wmd` adverse keyword group contained "arms trafficking" and "weapons smuggling" but not "arms dealer", "arms trade", "arms broker", "gunrunning", or 12 other related terms. Viktor Bout's news articles — "Viktor Bout, the arms dealer Russia has swapped for Brittney Griner" — returned `severity: "clear"` and `keywordGroups: []` despite him being listed on OFAC SDN. Any subject described in news as an "arms dealer" or "arms broker" will silently receive a false CLEAR adverse-media verdict.  
**Root cause:** Keyword list seeded from FATF typology vocabulary ("arms trafficking") rather than journalistic vocabulary ("arms dealer").  
**Fix applied:** Added 16 terms: `arms dealer`, `arms trade`, `arms broker`, `arms embargo`, `gunrunning`, `gun dealer`, `gun runner`, `gun trafficking`, `illicit arms`, `illegal arms`, `weapons dealer`, `weapons trade`, `weapons broker`, `small arms`, `light weapons`, `salw`, plus `munitions`, `explosives trafficking`, `military equipment`, `defense export`.  
**Verification:** Re-run `news_search` for "Viktor Bout" after deploy — `keywordGroups` should contain `proliferation-wmd` and `topSeverity` should be `critical`.

### 4.2 HIGH

#### H-1 — matchEnsemble stub type mismatch → silent fuzzy-score zeroing when dist/ not loaded
**File:** `web/app/api/news-search/route.ts:13-18`  
**Finding:** The fallback stub typed as `(a, b) => { score: number; method: string }` returned `{score, method}`. The call site accesses `m.best.score` — property `best` doesn't exist on the stub return, so `m.best` is `undefined` and the if-condition silently fails. When `dist/` is absent (local dev, partial build, cold Lambda before module init completes), ALL articles receive `fuzzyScore = 0` from the ensemble path, degrading to token-presence-only matching.  
**Root cause:** Type mismatch between stub return shape and `EnsembleMatch` interface defined in `src/brain/matching.ts`.  
**Fix applied:** Replaced stub return type with proper `EnsembleMatch` shape including `best: MatchScore`. Stub now returns `{ subject, candidate, scores: [score], best: score, phoneticAgreement: false }`.  
**Severity:** HIGH — degraded matching quality in dev/partial-build environments; production unaffected when `dist/` loads.

#### H-2 — NEXT_PUBLIC_APP_URL not set in environment
**Finding:** System status confirms `NEXT_PUBLIC_APP_URL` missing. This variable is used as the `Access-Control-Allow-Origin` CORS header in at least 10 API routes (`adverse-media-live`, `lei-lookup`, `vessel-check`, `domain-intel`, `mlro-advisor-challenger`, `ftm-export`, `benford`, `gleif`, `eval-kpi`, `domain-intel`). Fallback is hardcoded to `"https://hawkeye-sterling.netlify.app"`. If the deployment URL differs from that hardcoded value, all same-origin API calls from the browser will be rejected by CORS.  
**Root cause:** Environment variable not set in Netlify dashboard.  
**Fix required (operator action):** Set `NEXT_PUBLIC_APP_URL=https://<your-deployment-url>` in Netlify Site settings → Environment variables.  
**Risk:** CORS failure on the live deployment URL → UI completely non-functional for API calls.

#### H-3 — PEP corpus metric misleadingly reports "6" entries
**File:** `web/app/api/status/route.ts:943-946`  
**Finding:** Status metric defaulted to `Number(process.env["KNOWN_PEP_ENTRIES"] ?? "6")` — the literal number 6. The static `known-entities.ts` file has 75 PEP entries + additional adverse subjects. The warning "PEP corpus has only 6 entries — expected ≥500,000" alarmed operators and generated false urgency, obscuring the real coverage picture.  
**Root cause:** Default of 6 was a placeholder that should have been replaced with the actual static count.  
**Fix applied:** Status route now imports `KNOWN_PEPS` and `KNOWN_ADVERSE` from `known-entities.ts` and uses `Math.max(staticPepCount, Number(KNOWN_PEP_ENTRIES ?? "0"))` as the floor. Warning text updated to accurately describe coverage tiers.

#### H-4 — GOAML_MLRO_EMAIL and GOAML_MLRO_FULL_NAME absent from system_status config health
**File:** `web/app/api/status/route.ts:1051-1065`  
**Finding:** The `configHealth` section of `system_status` checked 13 environment variables but omitted `GOAML_MLRO_FULL_NAME` and `GOAML_MLRO_EMAIL` — both required for STR/SAR XML generation under UAE FDL 10/2025. An operator with these missing would not see a `configHealth.requiredMissing` alert, potentially submitting blank MLRO fields on every STR filing.  
**Root cause:** Config check list not kept in sync with `env-check/route.ts` which correctly marks both as required.  
**Fix applied:** Both variables added as `required: true` to `CONFIG_CHECKS` and `VAR_MAP` in `status/route.ts`.

### 4.3 MEDIUM

#### M-1 — GDELT live feed persistently degraded (10,492ms timeout, fetch failed)
**Finding:** `externalChecks` shows `gdelt-live-feed: degraded` with `latencyMs: 10492` and `note: "fetch failed"`. GDELT is used by both `adverse-media` and `weaponized-brain` routes. The `threatSurface` correctly flags this as `major` severity impairment of "Adverse media auto-detection (GDELT 10-year lookback, Art.19)" and "Weaponized Brain live news feed."  
**Root cause:** GDELT API intermittent availability; no circuit breaker with cached degraded result until recovery.  
**Status:** Previously noted fix (`fix(status): cache GDELT degraded results`) applied but GDELT is currently unreachable at the GDELT endpoint. External dependency — no code fix available.  
**Required action:** Confirm GDELT endpoint URL is correct in environment; if GDELT permanently unavailable, evaluate replacing with paid adverse-media vendor.

#### M-2 — Sanctions lists: zero records loaded (null ageH / null recordCount)
**Finding:** All 7 sanctions lists report `ageH: null, recordCount: null`. System indicates "awaiting first scheduled refresh" at 03:00 UTC 2026-05-13. Until refresh runs, the screening engine relies on in-memory seed corpus only.  
**Root cause:** New deployment; scheduled function hasn't run yet.  
**Required action:** Confirm `SANCTIONS_CRON_TOKEN` is set (✅ confirmed), confirm scheduled functions are enabled in Netlify dashboard, and wait for 03:00 refresh. Alternatively, trigger manually via `POST /api/sanctions/watch`.

#### M-3 — Circuit breaker state is in-memory (lost on Lambda cold start)
**File:** `web/lib/server/circuitBreaker.ts`  
**Finding:** `const breakers = new Map<string, BreakerState>()` — module-level, not persisted. On every cold start, all circuit-breaker state resets. A service that was failing before the cold start re-enters "closed" state and will make a full burst of requests against the failing upstream before the breaker re-opens.  
**Risk:** Low-medium — cold starts are infrequent on Netlify Pro with warm pool; most production impact is limited to the first few requests post-cold-start.  
**Recommended fix:** Persist circuit-breaker state in Netlify Blobs or (preferred) Upstash Redis with a short TTL.

#### M-4 — Rate limiter race condition (no atomic compare-and-swap)
**File:** `web/lib/server/rate-limit.ts:1-10`  
**Finding:** Documented in code: concurrent requests can both read count=N, both pass, both write count=N+1, effectively allowing N+2. Upstash Redis (which would fix this) is not configured.  
**Risk:** Low at current traffic levels; at high concurrency, burst-window enforcement is probabilistic.  
**Recommended fix:** Set `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` in Netlify env.

#### M-5 — TypeScript and ESLint errors suppressed at build
**File:** `web/next.config.mjs:9-17`  
**Finding:** `ignoreBuildErrors: true` and `ignoreDuringBuilds: true` remain in place. Type errors and lint violations silently reach production.  
**Root cause:** Pre-existing JSX implicit-any errors (TS7026/TS2741) and a react-hooks stack overflow on `MlroAdvisorPage`.  
**Recommended fix:** Run `npm run build 2>&1 | grep "error TS"` to enumerate errors; fix the stack-overflow component; re-enable strict build checking.

#### M-6 — Brain catalogue last reviewed 2026-04-01 (>30 days ago)
**Finding:** `feedVersions.reviewedAt = "2026-04-01"` — 41 days old at audit time. Status warning fires correctly. MLRO should trigger a catalogue review cycle.  
**Required action:** Set `BRAIN_REVIEWED_AT` env var to today's date after completing review; or update the hardcoded date in `status/route.ts:942`.

#### M-7 — Transient 502 errors on AI-powered MCP tools
**Finding:** During live testing, `screen_subject`, `sanctions_status`, and `pep_profile` MCP tools all returned `502 Bad Gateway` from Cloudflare (zone: `api.anthropic.com`). Error message: "origin web server returned an invalid or incomplete response." Retry guidance: wait 60 seconds.  
**Root cause:** Anthropic API temporarily overloaded at time of test. The MCP tools propagate the raw HTTP error rather than returning a structured fallback.  
**Risk:** During Anthropic outages, all AI-powered MCP tools become unavailable with opaque error messages.  
**Recommended fix:** Add `withLlmFallback` (already exists in `web/lib/server/llm-fallback.ts`) to MCP tool handlers, or add retry logic with exponential backoff within the tool.

### 4.4 LOW / INFO

#### L-1 — adverseMediaKeywords count in status (1066) is stale after this session's additions
**Finding:** `feedVersions.adverseMediaKeywords` is hardcoded to 1066 in `status/route.ts`. This session added ~20 new terms to `adverse-keywords.ts`.  
**Fix:** Count dynamically: `ADVERSE_KEYWORDS.reduce((n, r) => n + r.terms.length, 0)`.

#### L-2 — knownPepEntries metric now uses static count (75) which is still far below 500K threshold
**Finding:** After fix H-3, the PEP count is 75 (static list), still triggering the `< 500,000` warning. This is accurate — without LSEG World-Check or OpenSanctions Pro, live PEP coverage is genuinely limited. The warning is now correct and informative.  
**Required action:** Set `LSEG_WORLDCHECK_API_KEY` or `OPENSANCTIONS_PRO_API_KEY` for full PEP corpus coverage.

#### L-3 — Service worker (`/sw.js`) caches API responses without checking compliance-specific headers
**Finding:** `public/sw.js` implements a basic service worker for PWA. No review of whether compliance-critical API responses (screening verdicts, case data) could be stale-served from SW cache.  
**Risk:** Negligible if SW uses network-first strategy; needs verification.

#### L-4 — WEBHOOK_SECRET absent → outbound events unsigned
**Finding:** `web/lib/server/webhook.ts` logs `console.warn` when `HAWKEYE_WEBHOOK_SECRET` is missing. Outbound webhook events can be spoofed by any receiver if no secret is set.  
**Required action (if webhooks used):** Set `HAWKEYE_WEBHOOK_SECRET` and verify HMAC signature on receiver.
