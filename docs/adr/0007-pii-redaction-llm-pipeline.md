# ADR-0007: PII redaction in the LLM pipeline

- **Status:** Accepted
- **Date:** 2026-07-05
- **Deciders:** MLRO, Engineering Lead, Data Science Lead
- **Governance-impacting:** Yes
- **Regulatory anchor:** UAE FDL 45/2021 (PDPL); GDPR (data minimisation); UAE FDL 10/2025

## Context

Compliance narratives are generated by an external LLM. Sending raw customer PII
to a third-party model would breach data-minimisation and PDPL/GDPR obligations
and expand the disclosure surface. Yet the model still needs enough structure to
produce a useful, accurate narrative, and the returned text must read naturally
with the real entities restored inside our trust boundary.

## Decision

The LLM pipeline (`web/lib/server/llm.ts`) is **redact → model → rehydrate**:

1. **Redact** — PII is replaced with stable placeholders before the prompt
   leaves the process. Raw PII never crosses the application → provider boundary.
2. **Model** — the provider (per the model router, ADR-0006) sees only redacted,
   minimised input.
3. **Rehydrate** — placeholders are mapped back to real entities **inside** the
   trust boundary, so downstream consumers get the complete narrative.

Supporting controls:

- The **hallucination gate** runs fire-and-forget and must not block the response
  path (invariant #5); any await is wrapped in `void … .catch(…)`.
- No raw PII enters logs; the pipeline emits no C4 data to telemetry.
- The generated output still passes the **egress/tipping-off gate** before
  reaching any external sink (ADR-0002).

## Consequences

- The third-party model never receives raw PII — data-minimisation by
  construction, satisfying PDPL/GDPR.
- Rehydration keeps narrative quality high while confining plaintext PII to the
  trust boundary.
- The redaction map is sensitive (C3/C4-adjacent) and is confined to the request
  scope; it is never logged or persisted with the audit entry.
- A residual dependency on redaction completeness is accepted and tested; gaps
  are treated as security incidents.

## Alternatives considered

- **Send raw PII to the model** — simplest, but breaches data minimisation and
  expands disclosure; rejected outright.
- **Fully local model to avoid egress** — removes the third-party boundary but is
  not viable at the required quality/cost today; revisit if that changes.
- **Redact without rehydration** — safe but produces placeholder-laden narratives
  unusable for filings; rejected.
