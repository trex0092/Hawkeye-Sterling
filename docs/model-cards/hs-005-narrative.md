## 1. System Description

HS-005 generates structured STR (Suspicious Transaction Report) and SAR (Suspicious Activity Report) narrative skeletons for MLRO review and completion. It selects the appropriate skeleton template for the detected typology, injects verifiable observable facts from the HS-001/HS-002 pipeline output, runs the tipping-off guard, applies the compliance charter prohibitions, and returns a draft narrative ready for MLRO review.

HS-005 does **not** auto-submit reports. Submission via goAML requires explicit MLRO authorisation. Auto-submit capability exists at `POST /api/goaml/auto-submit` but is gated by the MLRO approval workflow and is disabled by default; activation requires separate MLRO sign-off.

The auto-submit endpoint enforces the following pre-submission gates:
1. MLRO-signed approval token (JWT, 4-hour TTL)
2. Tipping-off scan PASS (zero high-severity matches)
3. All mandatory sections populated (§3 below)
4. goAML XML schema validation PASS
5. Compliance charter P1–P10 check PASS

If any gate fails, the submission is blocked and the MLRO is returned a structured error with the specific failure reason.

## 3. Mandatory Seven-Section Narrative Structure

Every STR/SAR narrative generated by HS-005 must contain all seven sections. Missing sections trigger a `SECTION_INCOMPLETE` block on the draft card; the MLRO cannot submit until all sections are populated.

| Section | Title | Content Requirement |
|---|---|---|
| **§1** | Subject Profile | Full name, aliases, entity type, nationality/registration, identification documents, date of birth/incorporation, address(es) |
| **§2** | Relationship History | Duration of relationship, product(s)/service(s) used, onboarding date, prior STR/SAR/FFR/PNMR history (internal reference only — no disclosure to subject) |
| **§3** | Observable Facts | Chronological list of factual observations: transaction amounts, dates, counterparties, instruments — no conclusions, no allegations (charter P3 and P5) |
| **§4** | Typology Match | The specific typology(ies) from the Hawkeye Sterling taxonomy that the observable facts are consistent with; red-flag IDs cited; reasoning mode IDs cited |
| **§5** | Risk Assessment | MLRO's own risk conclusion: why the activity is considered suspicious; what the reasonable MLRO would conclude from the facts |
| **§6** | Actions Taken | Measures applied: EDD conducted, account restrictions, prior regulatory contact, inter-agency referrals |
| **§7** | MLRO Declaration | Date of suspicion formation; MLRO name, role, and signature; statement that the report is complete, accurate, and not a tipping-off risk |

## 4. Compliance Charter Prohibitions Enforced (P1–P10)

All ten charter prohibitions are enforced at draft-generation time. The following table maps the prohibitions most critical to narrative generation:

| Prohibition | ID | Enforcement in HS-005 |
|---|---|---|
| No unsourced sanctions assertions | P1 | Sanctions facts are injected only from the HS-001 list-match output; no training-data assertions |
| No fabricated adverse media | P2 | Only adverse-media items returned by HS-003 with source URLs are included; no generated citations |
| No legal conclusions | P3 | Narrative templates use "indicator of" / "typology match with" / "red flag for"; all conclusion phrasing is blocked |
| No tipping-off | P4 | Tipping-off guard runs on every draft; see §5 below |
| No allegation-to-finding upgrade | P5 | Template language enforces "alleged/reported/accused" for unproven claims |
| No identity merging | P6 | Each candidate is presented in a separate profile block |
| No clean result without scope declaration | P7 | Scope declaration is auto-populated in §1 |
| No training-data as current source | P8 | All citations trace to live pipeline inputs |
| No unexplained risk scores | P9 | §4 requires methodology, inputs, and weighting for every typology match |
| No proceeding on insufficient information | P10 | If mandatory §3 facts are absent, the draft is blocked with a structured gap list |

## 5. Tipping-Off Guardrails (FDL 10/2025 Art. X)

The tipping-off guard (`tipping-off-guard.ts`) runs on every narrative draft before it is displayed to the MLRO and again before goAML submission. The guard scans for:

| Pattern Category | Severity | Action |
|---|---|---|
| Explicit STR/SAR/FFR/PNMR reference in customer-facing egress | HIGH | Block egress — hard stop |
| Explicit goAML reference in customer-facing egress | HIGH | Block egress — hard stop |
| Ongoing investigation disclosure | HIGH | Block egress — hard stop |
| Filing acknowledgement to subject | HIGH | Block egress — hard stop |
| Regulator/FIU enquiry disclosure | HIGH | Block egress — hard stop |
| Instructions to move funds | HIGH | Block egress — hard stop |
| Suspicion-linked reason for offboarding | MEDIUM | Flag for MLRO review |
| AML-suspicion language in neutral communication | MEDIUM | Flag for MLRO review |

The guard **fails closed**: any HIGH-severity match results in a complete block. The MLRO is shown the matched excerpt, the rule ID, and a suggested neutral-language replacement. No sanitisation is performed automatically — the MLRO must rewrite the affected section.

> **Regulatory basis**: UAE Federal Decree-Law 20/2018 Art. 25 (as amended); FDL 10/2025 Art. X. Tipping-off is a criminal offence carrying penalties including imprisonment.

## 6. Human MLRO Review Requirement

**The MLRO must review, complete, and sign the narrative before any goAML submission.** HS-005 generates a skeleton; the MLRO is responsible for:

1. Verifying that §3 (Observable Facts) is complete, accurate, and consistent with the case file.
2. Completing §5 (Risk Assessment) — this section must contain the MLRO's own professional judgement and cannot be auto-generated.
3. Signing §7 (MLRO Declaration) with their registered goAML credentials.
4. Confirming that no information in the narrative would constitute tipping-off if subsequently disclosed in litigation.

The audit chain records the MLRO's identity, timestamp, and the hash of the final approved narrative. Any post-approval modification triggers a new approval cycle.

## 8. Quality Checks

HS-005 applies the following automated quality checks before releasing a draft for MLRO review:

| Check | Description | Failure Action |
|---|---|---|
| Section completeness | All 7 sections have non-empty content | Block draft; list missing sections |
| Tipping-off scan | Zero HIGH-severity matches | Block draft; show matched excerpts |
| Charter P3/P5 compliance | No legal conclusion or allegation-upgrade phrasing | Block draft; highlight offending phrases |
| Source citation validity | All factual claims trace to pipeline inputs with source URLs | Flag uncited claims |
| goAML schema validation | Draft XML is schema-valid | Block submission; show schema errors |
| MLRO signature token | JWT present, unexpired, issued to registered MLRO | Block submission |
| Duplicate submission check | Same subject + typology + date window not previously submitted | Warn MLRO; require explicit override |

## 10. Known Limitations

1. **MLRO §5 auto-generation is prohibited**: The risk assessment section must reflect MLRO professional judgement. Any attempt to auto-generate §5 text is blocked.
2. **Template coverage**: Emerging typologies not yet in the template library require the MLRO to use the generic `freeform` template, which does not pre-populate §4 typology references.
3. **Multilingual narrative generation**: HS-005 generates narratives in English only. goAML accepts Arabic and English submissions; Arabic narrative drafting requires manual translation.
4. **Historical data window**: Observable facts are drawn from the current pipeline run. Historical transaction data must be manually inserted by the MLRO from the case management system.
